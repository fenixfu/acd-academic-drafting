# 学术写作执行计划

> 基于 AGENTS.md 与 skills 定义，本计划为大步骤下的子步骤细化。

---

## 阶段零：启动准备

- 0.1 检查 `./output/` 目录是否有旧文件
- 0.2 若有文件，询问用户是否执行 `./backup_output.sh` 备份
- 0.3 检查 `./references/` 目录的文件
- 0.4 向用户确认文件对应关系（征稿启事 / 讨论记录）
- 0.5 加载 `academic-writer-persona` skill 激活执笔者人格

---

## 阶段一：征稿启事技术解析（cfp-analyzer 步骤一）

**输出文件**：`./output/01_技术规格书.md`

- 1.1 提取形式要求清单（字数/摘要/关键词/引用格式/匿名要求/提交格式/截止日期）
- 1.2 分析主题范围界定（明确议题 / 隐含理论偏好 / 边界案例）
- 1.3 推测目标期刊学术定位（审稿人画像 / 方法论偏好 / 引用生态）
- 1.4 识别写作禁忌（不欢迎话题 / 不接受方法 / 隐性禁忌）
- 1.5 预测评分权重分布（理论创新性 / 论证严密性 / 案例支撑等）
- 1.6 输出技术规格书
- 1.7 更新 `progress.md`
- 1.8 调用 subagent 执行断点检查

---

## 阶段二：讨论成果结构化整理（cfp-analyzer 步骤二）

**输出文件**：`./output/02_结构化素材库.md`

- 2.1 提取3–5个核心论点（含原始表述→学术表述转化）
- 2.2 分类支撑性论据（理论依据 / 主要案例 / 对照案例 / 数据）
- 2.3 识别潜在学术争议点
- 2.4 整理理论资源清单（含核实状态）
- 2.5 梳理逻辑链条与薄弱环节
- 2.6 输出结构化素材库
- 2.7 更新 `progress.md`
- 2.8 调用 subagent 执行断点检查

---

## 阶段三：文章结构学术化设计（cfp-analyzer 步骤三）

**输出文件**：`./output/03_写作方案.md`

- 3.1 拟定3个备选标题（主副标题结构，含质量评估）
- 3.2 用户选择标题（暂停交由用户决定）
- 3.3 根据字数确定结构层级（5000字内/5000-10000字/10000字以上）
- 3.4 设计引言结构（论证任务 / 关键操作 / 目标字数）
- 3.5 设计各节正文结构（论证任务 / 二级标题 / 目标字数）
- 3.6 设计结语结构（论证任务 / 禁止操作）
- 3.7 说明征稿对应、讨论落地、论证逻辑、风险提示
- 3.8 输出写作方案
- 3.9 更新 `progress.md`
- 3.10 调用 subagent 执行断点检查

---

## 阶段四：去AI化语言风格校准（academic-drafter 步骤四）

🔴 除去最后的断点检查，本阶段列出的主要任务**务必由主agent亲自完成**，辅助和检验型的任务才能委托subagent

**输出文件**：`./output/04_语言风格校准书.md`

- 4.1 生成禁用表达库（过渡套话10–15个 / 空洞概念10–15个 / 机械结构5–8种）
- 4.2 生成替代方案库（理论批判转折词15–20个 / 游戏研究术语20–30个 / 不规则句式10个）
- 4.3 设定语言人格参数（句子长度 / 从句频率 / 术语密度 / 引用节奏）
- 4.4 输出语言风格校准书
- 4.5 更新 `progress.md`
- 4.6 调用 subagent 执行断点检查

---

## 阶段五：分节写作（academic-drafter 步骤五）

🔴 本阶段列出的主要写作任务**务必由主agent亲自完成**，辅助和检验型的任务才能委托subagent

**输出文件**：`./output/05_初稿.md`

> **🔴 红线五：论证必须强制落地到具体案例**
> - 任何抽象论点和理论概念都必须能落地到具体的游戏文本、实际游玩体验或局内机制
> - **严禁**：空洞堆砌、生硬嫁接、停留在"呈现了X现象"的表层

- 5.1 向用户确认当前章节信息（字数/论证任务/理论资源/案例/引用格式）
- 5.2 执行引言写作
- 5.3 每写完一个自然段，调用 subagent 检查（禁用表达 / 案例支撑 / 论证推进 / 句式变化）
  - **🔴 红线二**：必须要求subagent激活 `academic-writer-persona` skill
- 5.4 根据检查报告立即修改
- 5.5 依次执行各节正文写作（重复 5.3–5.4 流程）
- 5.6 执行结语写作
- 5.7 每节完成后调用 subagent 评价（完成情况 / 遗留问题 / AI痕迹风险）
- 5.8 根据评价立即修改
- 5.9 输出初稿
- 5.10 更新 `progress.md`
- 5.11 调用 subagent 执行断点检查

---

## 阶段六：批判性回读与理论深化（academic-drafter 步骤六）

**输出文件**：`./output/06_问题清单.md`、`./output/06_修改稿.md`


🔴 本阶段的主要任务为根据要求进行修改，这部分任务**务必由主agent亲自完成**，辅助和检验型的任务才能委托subagent
> **🔴 红线一：禁止破坏暂停断点** - 本步骤末尾必须停下，委托 subagent 判断执行情况
> **🔴 红线二：必须让 Subagent 激活`academic-writer-persona`**

- 6.1 调用 subagent，要求其激活 `academic-writer-persona` 技能，并传递初稿路径
- 6.2 检查理论一致性
- 6.3 检查论证充分性
- 6.4 检查案例典型性
- 6.5 检查学术对话性
- 6.6 检查批判建设性
- 6.7 输出问题清单（含位置/问题/修改方向）
- 6.8 立刻根据问题清单修改
- 6.9 运行 `word_count.py` 统计字数，根据结果增删篇幅
- 6.10 输出修改稿
- 6.11 更新 `progress.md`
- 6.12 **🔴 暂停断点**：调用 subagent 执行断点检查，有疑虑则向用户确认

## 阶段七：AI痕迹最终扫描与内容深度打磨（academic-drafter 步骤七）

**输出文件**：`./output/07_扫描报告.md`、`./output/07_修改稿.md`

本阶段的主要任务为根据要求进行修改，这部分任务**务必由主agent亲自完成**，辅助和检验型的任务才能委托subagent
> **🔴 红线一：禁止破坏暂停断点** - 本步骤末尾必须停下，委托 subagent 判断执行情况
> **🔴 红线二：必须让 Subagent 激活`academic-writer-persona`**
> - 五项扫描（7.1–7.5）每项结果必须返回用户确认

- 7.1 扫描过渡句自然性，标记疑似AI过渡句，**结果返回用户确认**
- 7.2 检查结构有机性（各节长度比例），**结果返回用户确认**
- 7.3 确认论证曲折性（2–3处理论自我质疑），**结果返回用户确认**
- 7.4 检查引用个性化（教科书式引用序列），**结果返回用户确认**
- 7.5 确认语言温度（1–2处人的痕迹），**结果返回用户确认**
- 7.6 输出扫描报告
- 7.7 输出修改稿
- 7.8 更新 `progress.md`
- 7.9 **🔴 暂停断点**：调用 subagent 执行断点检查，有疑虑则向用户确认

## 阶段八：形式规范客观校对与定稿（academic-drafter 步骤八）

**输出文件**：`./output/08_格式检查报告.md`、`./output/08_最终稿.md`

本阶段的主要任务为根据要求进行修改，这部分任务**务必由主agent亲自完成**，辅助和检验型的任务才能委托subagent
> **🔴 红线一：禁止破坏暂停断点** - 本步骤末尾必须停下，委托 subagent 判断执行情况
> **🔴 红线二：必须让 Subagent 激活`academic-writer-persona`**
> **🔴 红线四：格式检查报告必须中性客观**
> - 对照步骤一的技术规格书"逐项检查，零容忍"
> - **严禁**：附加主观评价（如"完美契合标准"、"达到顶刊水平"、"适合立刻投稿"）

- 8.1 运行 `word_count.py` 统计字数，对照技术规格书判定是否合格
- 8.2 检查格式一致性（标题层级/段落格式/引号/数字）
- 8.3 检查引用规范（注释格式/参考文献格式/DOI-URL）
- 8.4 检查匿名性（正文/脚注/文件属性/致谢）
- 8.5 检查摘要与关键词（字数/结构/数量/检索词）
- 8.6 输出格式检查报告（**零容忍逐项检查，中性客观**）
- 8.7 立刻根据检查报告修改
- 8.8 输出投稿前信心度评估
- 8.9 输出最终稿
- 8.10 更新 `progress.md`
- 8.11 **🔴 暂停断点**：调用 subagent 执行断点检查，有疑虑则向用户确认

## 阶段九：进一步迭代（academic-drafter 步骤九）

**输出文件**：`./output/09_迭代稿_v[N].md`

> **🔴 红线三：严格执行版本控制更新**
> - 每次迭代**必须且只能**在文件头部添加或更新版本说明区
> - **严禁**：不建立或不更新头部档案的情况下连续覆盖写入

- 9.1 评估信心度（若<★★★★★的维度≥2个，重新执行步骤六、七）
- 9.2 询问用户是否有调整需求（格式/风格/逻辑/表达）
- 9.3 执行迭代修改
- 9.4 **🔴 必须更新**文件头部版本说明：
  ```
  版本：v[N]
  修改日期：[日期]
  本轮主要修改：[简述]
  ```
- 9.5 输出迭代稿
- 9.6 更新 `progress.md`
- 9.7 调用 subagent 执行断点检查
