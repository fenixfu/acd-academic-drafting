# 学术论文写作执行器（Academic Drafter）

> **前置检查**：本 skill 启动前，确认以下输入已就绪：
> - [ ] `academic-writer-persona` skill 已激活（人格底座）
> - [ ] `cfp-analyzer` 已完成步骤一（技术规格书）和步骤三（结构方案）
> - [ ] 目标文体、字数限定、引用格式、截止日期已明确
>
> 如果上述输入缺失，请先要求用户补充。



## 步骤四：去AI化语言风格校准

**触发时机**：在写作任何正文之前，必须先完成本步骤。

### 任务内容

在正式写作前，进行语言风格的"负面清单"检查。
根据输出结构中的项目清单，生成本文的"禁用表达库"和"替代方案库"：

### 输出结构

禁用表达（AI典型痕迹）：
- 过渡套话：
	- 【列举10-15个】
- 空洞概念：
	- 【列举10-15个】
- 机械结构：
	- 【列举5-8种】

其他禁忌：
- 禁止重复或直接引述征稿启事（CfP）中的表述
- 禁止重复或直接引述征稿启事（CfP）中的表述
- 禁止无法用证据支持的过度修辞的论断

替代方案（学术自然表达）：
- 理论批判类转折词：
	- 【提供15-20个】
- 游戏研究专业术语：
	- 【提供20-30个】
- 不规则论证句式：
	- 【提供10个示例】

设定本文的"语言人格"：
- 句子平均长度范围【默认30-50字】
- 从句使用频率【默认1句/200字】
- 专业术语密度【默认1个/句】
- 引用出现节奏【默认1个/250字】

### 收尾
**确认已将结果输出到文件**
**确认已经更新进度**
**执行断点检查**：输出当前步骤执行结果报告，然后将相应的上下文（包括：`plan.md`、对应的技能）和执行结果报告注入subagent，令其判断执行过程是否符合程序。如subagent认为有不符合流程或存在疑虑之处，**必须**向用户提问确认，**严禁**自作主张。

## 步骤五：分节写作

**触发条件**：用户指定当前要写的章节（如「引言」「第二节」）。

**对每一节，执行以下流程：**

### 5.1 写作前确认

在开始写作前，向用户确认：

```
当前章节：[节名]
目标字数：[X]字（字数限制的95%±5%，即[X-10%]至[X]字）
字数统计范围：[是/否]包含注释/参考文献
论证任务：[该节需要完成的核心论证步骤]
理论资源：计划引用[X]处，涉及[学者名]
案例：计划使用[游戏名]，切入角度为[机制/叙事/社区/美学]
引用格式：[APA/MLA/Chicago/GB/T等]
注释方式：[脚注/尾注]
```

### 5.2 写作原则

**强制要求：**
- 每个「因为/由于/鉴于」后的推理必须经得起反驳
- 每个案例必须有机制层面的具体分析（不得停留在「X游戏呈现了Y现象」）
- 每节至少有1处「思考的停顿」，例如：
  - 「这里或许需要更精确地区分……」
  - 「当然，这一解释也面临……的挑战」
  - 「另一种可能的解读路径是……」

**引用处理规范：**
- 直接引用：须标注页码
- 转述引用：须说明与原文的转化关系
- 批判性引用（引用是为了反驳）：须明确说明反驳立场
- 禁止连续两个段落都以他人引用收尾

**每写完一个自然段，执行一次检查：**
调用subagent，将`academic-writer-persona`skill全文作为系统提示词注入subagent，令其**中立客观地**检查以下项目
- [ ] 有无步骤四「禁用表达库」中的词汇？
- [ ] 上一个抽象论点是否有具体游戏文本支撑？
- [ ] 本段是否在重复上一段的论证，而非推进？
- [ ] 最近三句话的句式结构是否过于相同？
收到检查报告后，**严禁附加主观执行情况判断**，**立即**根据检查报告本身修改内容。

### 5.3 写作后评价

**触发条件**：每节完成后（如果是短文体，比如会议摘要等1000字以内的文稿，则在全文完成后执行）
**执行操作**：
- 调用subagent，将`academic-writer-persona`skill全文作为系统提示词注入subagent
- 注入该小节内容，令subagent按照如下格式输出对其的评价（100字左右）

```
本节完成情况：
✓ 已完成：[核心论证任务]
～ 部分完成：[哪些内容有待深化]
✗ 遗留问题：[哪些案例/引用需要核实]
AI痕迹风险：[本节最可能被识别为AI写作的句子/段落，建议修改方向]
```
收到检查报告后，**严禁附加主观执行情况判断**，**立即**根据检查报告本身修改内容。

### 5.4 收尾
**确认已将结果输出到文件**
**确认已经更新进度**
**执行断点检查**：输出当前步骤执行结果报告，然后将相应的上下文（包括：`plan.md`、对应的技能）和执行结果报告注入subagent，令其判断执行过程是否符合程序。如subagent认为有不符合流程或存在疑虑之处，**必须**向用户提问确认，**严禁**自作主张。

## 步骤六：批判性回读与理论深化

**触发条件**：初稿完成后立刻执行
**执行操作**：
- 调用subagent，将`academic-writer-persona`skill全文作为系统提示词注入subagent
- 向subagent传递初稿文件路径
- 令其根据批判维度列出的条目检查初稿，并根据输出格式示例输出问题清单

### 批判维度
1. 理论一致性：各段落的理论立场是否连贯？有无自相矛盾？
2. 论证充分性：每个"因此""可见""由此"之前的推理是否经得起推敲？
3. 案例典型性：游戏实例的选择是否具有代表性？是否需要补充反例？
4. 学术对话性：是否体现出对既有研究的回应和推进？
5. 批判的建设性：批判之后是否提出了替代性解释或新的理论可能？


### 输出格式

```
问题清单（按优先级排列）：
[问题1]：位置 → 问题描述 → 修改方向
[问题2]：...

需要补充的文献/案例：
[1] [文献/案例描述] → [在何处插入] → [承担何种论证功能]

建议深化的理论方向：
[描述] → 深化修改方向
```
### 收尾

- **必须立刻**根据问题清单做出修改。
- **必须立刻**运行word_count.py字数统计，根据结果增删篇幅。增删篇幅时注意可以使用grep_search和replace工具做目标明确的替换，以精准控制篇幅。

**确认已将结果输出到文件**
**确认已经更新进度**
**执行断点检查**：输出当前步骤执行结果报告，然后将相应的上下文（包括：`plan.md`、对应的技能）和执行结果报告注入subagent，令其判断执行过程是否符合程序。如subagent认为有不符合流程或存在疑虑之处，**必须**向用户提问确认，**严禁**自作主张。

## 步骤七：AI痕迹最终扫描与内容深度打磨

**触发条件**：步骤六批判性回读已完成，初稿已修改。

**执行原则**：
- 每一步单独调用subagent，将`academic-writer-persona`skill全文作为系统提示词注入subagent，并向subagent传递初稿文件路径
- 每一步结果必须返回给用户确认

### 扫描清单

#### 7.1 过渡句的自然性

通读全文，标记所有**疑似AI过渡句**：

- 句子本身不承载信息，只起「章节连接」功能
- 可以删去而不损失任何论证内容
- 操作：直接删除，或改写为承载信息的实质句
- 结果返回给用户确认

#### 7.2 结构的有机性

检查各节长度：

- 有无某节明显过短（不足计划字数的80%），暗示论证发育不足？
- 有无某节明显过长，导致节奏失控？
- 如有必要，**调整段落归属**（将某些段落从一节移到另一节）而非填充字数
- 结果返回给用户确认

#### 7.3 论证的曲折性

确认全文至少有**2–3处「理论的自我质疑」**：

- 不是虚假的谦虚，而是真正指出论证的边界或假设
- 位置建议：通常在主要论证节的末尾，或结语的开头
- 结果返回给用户确认

#### 7.4 引用的个性化

检查是否有**「教科书式引用序列」**：

- 症状：A提出概念→B发展概念→C批评概念→本文综合（只是介绍学术史，无自身立场）
- 治疗：将其中至少一处引用转为批判性引用，表明本文的倾向
- 结果返回给用户确认

#### 7.5 语言的温度

确认全文至少有**1–2处「人的痕迹」**：

- 审慎的个人判断（「在本文看来，……是更有解释力的路径」）
- 对学术共同体的真实致敬（不是套话感谢，而是真正指出某位学者的贡献对本文的启发）
- 对游戏本身的智识热情（允许偶尔出现的非中立语气）
- 结果返回给用户确认

### 扫描报告输出

```
AI痕迹扫描报告：
调整了 [X] 处过渡句（已删除/改写）
调整了 [X] 处结构（具体说明）
新增/强化了 [X] 处理论自我质疑
改写了 [X] 处教科书式引用序列
确认了 [X] 处人的痕迹（位置说明）
```
### 收尾
**确认已将结果输出到文件**
**确认已经更新进度**
**执行断点检查**：输出当前步骤执行结果报告，然后将相应的上下文（包括：`plan.md`、对应的技能）和执行结果报告注入subagent，令其判断执行过程是否符合程序。如subagent认为有不符合流程或存在疑虑之处，**必须**向用户提问确认，**严禁**自作主张。

## 步骤八：形式规范客观校对与定稿

**触发条件**：步骤七AI痕迹扫描已通过。

**执行原则**：
- 每一步单独调用subagent，传递步骤七修改过的文件路径
- 让subagent对照步骤一的输出文件（技术规格书），**中立客观地逐项检查，严格遵循标准**。
- 收到subagent的检查报告后，**严禁在客观结果之上附加主观执行情况判断**，立即根据检查报告本身修改文件。

### 检查清单

#### 8.1 字数统计

运行word_count.py脚本统计字数。
读取步骤一的输出文件（技术规格书），确认其中的字数要求（包括是否有弹性），**严格遵循标准**判定是否合格。

#### 8.2 格式一致性

- [ ] 标题层级是否统一（字号、加粗、编号方式）
- [ ] 段落格式是否规范（首行缩进2字符 or 段前空行，两者不混用）
- [ ] 引号使用是否统一（中文用「」或""，全文一致）
- [ ] 数字格式是否统一（全角/半角，阿拉伯数字/汉字）

#### 8.3 引用规范

**注释格式检查（每条逐一核对）：**
- [ ] 作者姓名格式是否一致（中文姓名/西文姓名倒置规则）
- [ ] 年份位置是否符合指定格式
- [ ] 页码是否标注（直接引用必须有页码）
- [ ] 出版信息是否完整（出版社/期刊名/卷期号）
- [ ] 在线资源是否标注访问日期

**参考文献格式检查：**
- [ ] 是否按征稿要求排序（字母/引用顺序/中英分开）
- [ ] 中英文文献是否按要求分开列出
- [ ] DOI/URL是否完整

#### 8.4 匿名性检查（双盲投稿）

- [ ] 正文中是否出现作者姓名
- [ ] 脚注/尾注中是否有自引（「见拙作……」类表述）
- [ ] 文件属性中是否已删除作者信息
- [ ] 致谢部分是否已移除（如征稿要求匿名）

#### 8.5 摘要与关键词

- [ ] 摘要字数符合要求
- [ ] 摘要是否包含：研究问题、方法/路径、核心论点、理论贡献
- [ ] 关键词数量符合要求（通常3–8个）
- [ ] 关键词是否包含目标检索词（领域术语而非自创概念）

### 格式检查报告输出

```
形式规范检查报告

整体状态：[通过 / 有待修正]

不通过项（需修改后重新提交）：
[项目名]：[具体问题] → [修改指令]

建议优化项（不影响投稿但建议改进）：
[项目名]：[建议]
```

### 投稿前信心度评估

```
投稿前信心度评估：

形式合规性：[95–100% / 90–95% / 低于90%]（低于90%需重做步骤八）
理论创新性：★★★☆☆ [说明判断依据]
论证严密性：★★★★☆ [说明]
案例支撑力：★★★☆☆ [说明]
去AI化程度：★★★★☆ [说明残留风险]
与征稿匹配度：[高/中/低] [说明]

综合预期：[强烈推荐投稿 / 建议小修后投稿 / 建议重大修改后投稿]
主要风险点：[1–2个最可能被审稿人质疑的地方]
加分亮点：[1–2个最可能打动审稿人的地方]
```
### 收尾
**确认已将结果输出到文件**
**确认已经更新进度**
**执行断点检查**：输出当前步骤执行结果报告，然后将相应的上下文（包括：`plan.md`、对应的技能）和执行结果报告注入subagent，令其判断执行过程是否符合程序。如subagent认为有不符合流程或存在疑虑之处，**必须**向用户提问确认，**严禁**自作主张。

## 步骤九：进一步迭代

「投稿前信心度评估」维度中如果有2个以上达不到★★★★★，则重新进行步骤六（批判性回读）和步骤七（AI痕迹扫描），直到「投稿前信心度评估」所有维度均为★★★★★（允许有一个★★★★☆），再提交。

**用户主动要求迭代时，询问用户是否有以下需求：**
- 格式调整
- 风格调整
- 逻辑调整
- 不合适的表达
