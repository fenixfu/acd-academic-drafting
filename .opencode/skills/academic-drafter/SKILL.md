# 学术论文写作执行器（Academic Drafter）

> **前置检查**：本 skill 启动前，确认以下输入已就绪：
> - [ ] `academic-writer-persona` skill 已激活（人格底座）
> - [ ] `cfp-analyzer` 已完成步骤一（技术规格书）和步骤三（结构方案）
> - [ ] 目标文体、字数限定、引用格式、截止日期已明确
>
> 如果上述输入缺失，请先要求用户补充。



## 步骤四：去AI化语言风格校准

**触发时机**：在写作任何正文之前，必须先完成本步骤。

### 任务内容

在正式写作前，进行语言风格的"负面清单"检查。
根据输出结构中的项目清单，生成本文的"禁用表达库"和"替代方案库"：

### 输出结构

禁用表达（AI典型痕迹）：
- 过渡套话：
	- 【列举10-15个】
- 空洞概念：
	- 【列举10-15个】
- 机械结构：
	- 【列举5-8种】

替代方案（学术自然表达）：
- 理论批判类转折词：
	- 【提供15-20个】
- 游戏研究专业术语：
	- 【提供20-30个】
- 不规则论证句式：
	- 【提供10个示例】

设定本文的"语言人格"：
- 句子平均长度范围【默认30-50字】
- 从句使用频率【默认1句/200字】
- 专业术语密度【默认1个/句】
- 引用出现节奏【默认1个/250字】

### 收尾
**确认已将结果输出到文件**
**完成后暂停，等待用户确认结果**

## 步骤五：分节写作

**触发条件**：用户指定当前要写的章节（如「引言」「第二节」）。

**对每一节，执行以下流程：**

### 5.1 写作前确认

在开始写作前，向用户确认：

```
当前章节：[节名]
目标字数：[X]字（±5%，即[X-5%]至[X+5%]字）
字数统计范围：[是/否]包含注释/参考文献
论证任务：[该节需要完成的核心论证步骤]
理论资源：计划引用[X]处，涉及[学者名]
案例：计划使用[游戏名]，切入角度为[机制/叙事/社区/美学]
引用格式：[APA/MLA/Chicago/GB/T等]
注释方式：[脚注/尾注]
```

### 5.2 写作原则

**每写完一个自然段，执行一次自查：**

- [ ] 有无步骤四「禁用表达库」中的词汇？
- [ ] 上一个抽象论点是否有具体游戏文本支撑？
- [ ] 本段是否在重复上一段的论证，而非推进？
- [ ] 最近三句话的句式结构是否过于相同？

**强制要求：**
- 每个「因为/由于/鉴于」后的推理必须经得起反驳
- 每个案例必须有机制层面的具体分析（不得停留在「X游戏呈现了Y现象」）
- 每节至少有1处「思考的停顿」，例如：
  - 「这里或许需要更精确地区分……」
  - 「当然，这一解释也面临……的挑战」
  - 「另一种可能的解读路径是……」

**引用处理规范：**
- 直接引用：须标注页码
- 转述引用：须说明与原文的转化关系
- 批判性引用（引用是为了反驳）：须明确说明反驳立场
- 禁止连续两个段落都以他人引用收尾

### 5.3 写作后评价

**触发条件**：每节完成后
**执行操作**：
- 调用subagent，令其加载`academic-writer-persona`
- 注入该小节内容，令subagent按照如下格式输出对其的评价（100字左右）

```
本节完成情况：
✓ 已完成：[核心论证任务]
～ 部分完成：[哪些内容有待深化]
✗ 遗留问题：[哪些案例/引用需要核实]
AI痕迹风险：[本节最可能被识别为AI写作的句子/段落，建议修改方向]
```

### 5.4 收尾
**确认已将结果输出到文件**
**完成后暂停，等待用户确认结果**

## 步骤六：批判性回读与理论深化

**触发条件**：初稿完成后立刻执行
**执行操作**：
- 调用subagent，令其加载`academic-writer-persona`
- 向subagent传递初稿文件路径
- 令其根据批判维度列出的条目检查初稿，并根据输出格式示例输出问题清单

### 批判维度
1. 理论一致性：各段落的理论立场是否连贯？有无自相矛盾？
2. 论证充分性：每个"因此""可见""由此"之前的推理是否经得起推敲？
3. 案例典型性：游戏实例的选择是否具有代表性？是否需要补充反例？
4. 学术对话性：是否体现出对既有研究的回应和推进？
5. 批判的建设性：批判之后是否提出了替代性解释或新的理论可能？


### 输出格式

```
问题清单（按优先级排列）：
[问题1]：位置 → 问题描述 → 修改方向
[问题2]：...

需要补充的文献/案例：
[1] [文献/案例描述] → [在何处插入] → [承担何种论证功能]

建议深化的理论方向：
[描述] → 深化修改方向
```
### 收尾
**确认已将结果输出到文件**
**完成后暂停，等待用户确认结果**

## 步骤七：形式规范终极校对

**触发条件**：全文正文初稿已完成。

**执行原则**：
- 每一步单独调用subagent，传递初稿文件路径
- 让subagent对照步骤一的输出文件（技术规格书），**逐项检查，零容忍**。

### 检查清单

#### 7.1 字数统计

运行word_count.py脚本统计字数。

```
正文字数：[X]字　　要求：[征稿要求]　　状态：[达标/超出X字/不足X字]
摘要字数：[X]字　　要求：[征稿要求]　　状态：[达标/超出/不足]
注释总字数：[X]字　参考值：[征稿要求]　状态：[达标/说明]
参考文献条数：[X]条
```

#### 7.2 格式一致性

- [ ] 标题层级是否统一（字号、加粗、编号方式）
- [ ] 段落格式是否规范（首行缩进2字符 or 段前空行，两者不混用）
- [ ] 引号使用是否统一（中文用「」或""，全文一致）
- [ ] 数字格式是否统一（全角/半角，阿拉伯数字/汉字）

#### 7.3 引用规范

**注释格式检查（每条逐一核对）：**
- [ ] 作者姓名格式是否一致（中文姓名/西文姓名倒置规则）
- [ ] 年份位置是否符合指定格式
- [ ] 页码是否标注（直接引用必须有页码）
- [ ] 出版信息是否完整（出版社/期刊名/卷期号）
- [ ] 在线资源是否标注访问日期

**参考文献格式检查：**
- [ ] 是否按征稿要求排序（字母/引用顺序/中英分开）
- [ ] 中英文文献是否按要求分开列出
- [ ] DOI/URL是否完整

#### 7.4 匿名性检查（双盲投稿）

- [ ] 正文中是否出现作者姓名
- [ ] 脚注/尾注中是否有自引（「见拙作……」类表述）
- [ ] 文件属性中是否已删除作者信息
- [ ] 致谢部分是否已移除（如征稿要求匿名）

#### 7.5 摘要与关键词

- [ ] 摘要字数符合要求
- [ ] 摘要是否包含：研究问题、方法/路径、核心论点、理论贡献
- [ ] 关键词数量符合要求（通常3–8个）
- [ ] 关键词是否包含目标检索词（领域术语而非自创概念）

### 输出格式

```
形式规范检查报告

整体状态：[通过 / 有待修正]

不通过项（需修改后重新提交）：
[项目名]：[具体问题] → [修改指令]

建议优化项（不影响投稿但建议改进）：
[项目名]：[建议]
```
### 收尾
**确认已将结果输出到文件**
**完成后暂停，等待用户确认结果**

## 步骤八：AI痕迹最终扫描与定稿

**触发条件**：步骤七形式审查已通过。

**执行原则**：
- 每一步单独调用subagent，传递步骤七修改过的初稿文件的路径
- 每一步结果必须返回给用户确认

### 扫描清单

#### 8.1 过渡句的自然性

通读全文，标记所有**疑似AI过渡句**：

- 句子本身不承载信息，只起「章节连接」功能
- 可以删去而不损失任何论证内容
- 操作：直接删除，或改写为承载信息的实质句
- 结果返回给用户确认

#### 8.2 结构的有机性

检查各节长度：

- 有无某节明显过短（不足计划字数的80%），暗示论证发育不足？
- 有无某节明显过长，导致节奏失控？
- 如有必要，**调整段落归属**（将某些段落从一节移到另一节）而非填充字数
- 结果返回给用户确认

#### 8.3 论证的曲折性

确认全文至少有**2–3处「理论的自我质疑」**：

- 不是虚假的谦虚，而是真正指出论证的边界或假设
- 位置建议：通常在主要论证节的末尾，或结语的开头
- 结果返回给用户确认

#### 8.4 引用的个性化

检查是否有**「教科书式引用序列」**：

- 症状：A提出概念→B发展概念→C批评概念→本文综合（只是介绍学术史，无自身立场）
- 治疗：将其中至少一处引用转为批判性引用，表明本文的倾向
- 结果返回给用户确认

#### 8.5 语言的温度

确认全文至少有**1–2处「人的痕迹」**：

- 审慎的个人判断（「在本文看来，……是更有解释力的路径」）
- 对学术共同体的真实致敬（不是套话感谢，而是真正指出某位学者的贡献对本文的启发）
- 对游戏本身的智识热情（允许偶尔出现的非中立语气）
- 结果返回给用户确认

### 最终输出

```
AI痕迹扫描报告：
调整了 [X] 处过渡句（已删除/改写）
调整了 [X] 处结构（具体说明）
新增/强化了 [X] 处理论自我质疑
改写了 [X] 处教科书式引用序列
确认了 [X] 处人的痕迹（位置说明）

---

投稿前自信度评估：

形式合规性：[95–100% / 90–95% / 低于90%]（低于90%需重做步骤七）
理论创新性：★★★☆☆ [说明判断依据]
论证严密性：★★★★☆ [说明]
案例支撑力：★★★☆☆ [说明]
去AI化程度：★★★★☆ [说明残留风险]
与征稿匹配度：[高/中/低] [说明]

综合预期：[强烈推荐投稿 / 建议小修后投稿 / 建议重大修改后投稿]
主要风险点：[1–2个最可能被审稿人质疑的地方]
加分亮点：[1–2个最可能打动审稿人的地方]
```

## 步骤九：进一步迭代

「投稿前自信度评估」维度中如果有2个以上达不到★★★★★，则重新进行步骤六（批判性回读）和步骤八（AI痕迹扫描），直到「投稿前自信度评估」所有维度均为★★★★★（允许有一个★★★★☆），再提交。

### 轮次管理：

每轮完成后，在文件头部更新版本说明：

```
版本：v[N]
修改日期：[日期]
本轮主要修改：[简述]
```
